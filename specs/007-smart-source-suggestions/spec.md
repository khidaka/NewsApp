# Feature Specification: 利用統計に基づくおすすめソースの自動調整

**Feature Branch**: `007-smart-source-suggestions`

**Created**: 2026-05-29

**Status**: Draft

**Input**: User description: "おすすめニュースソースのリストを、ユーザーの利用統計（追加済みソース、記事のスキップ・共有・興味なしの傾向）に基づいて自動的に並び替え・選定するようにしたい。ローカルのデータのみを使用し、リモート取得や手動編集は行わない。現在の静的な7件リストをベースに、ユーザーの行動に合わせて表示順や優先度を調整する。"

## Clarifications

### Session 2026-05-29

- Q: おすすめソース（未追加）の優先度スコアを何で算出するか → A: 各おすすめソースに代表キーワード群を付与し、ユーザーのキーワード別シグナル重み（共有+2/興味なし-3）の合計でスコア化
- Q: スコアが同点の場合の並び順 → A: 現行の静的リスト順（コード定義順）をフォールバックにする

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 利用傾向に合ったおすすめが上位に表示される (Priority: P1)

ユーザーがソース追加シートを開くと、おすすめリストが自分の利用傾向（よく共有する／興味なしにする記事のジャンルやソース）に合わせて並び替えられ、興味に近いソースが上位に表示される。

**Why this priority**: 現状のおすすめは固定順で、ユーザーの好みと無関係。利用傾向に合わせて並べ替えることが本フィーチャーの核心的価値。

**Independent Test**: 特定ジャンルの記事を共有・興味なしにする操作を行った後、ソース追加シートを開き、おすすめの並び順がその傾向を反映していることを確認することで単独でテスト可能。

**Acceptance Scenarios**:

1. **Given** ユーザーがあるジャンル（例: テック系）の記事を繰り返し共有している, **When** ソース追加シートを開く, **Then** そのジャンルに対応するおすすめソースが上位に表示される
2. **Given** ユーザーがあるジャンルの記事を繰り返し「興味なし」にしている, **When** ソース追加シートを開く, **Then** そのジャンルに対応するおすすめソースが下位に表示される
3. **Given** ユーザーが利用履歴をまだ持っていない（新規）, **When** ソース追加シートを開く, **Then** おすすめは既定の順序（現行の静的順）で表示される

---

### User Story 2 - 追加済みソースは引き続き除外される (Priority: P2)

並び替え後も、すでに追加済みのソースはおすすめリストに表示されない（feature 005 の重複防止を維持）。

**Why this priority**: 既存のワンタップ追加・重複防止の体験を壊さないことが重要。

**Independent Test**: いくつかのソースを追加した後、おすすめリストにそれらが含まれず、かつ残りが利用傾向順で並んでいることを確認することで単独でテスト可能。

**Acceptance Scenarios**:

1. **Given** ユーザーがおすすめの中のいくつかを追加済み, **When** ソース追加シートを開く, **Then** 追加済みソースは表示されず、残りが利用傾向順で並ぶ

---

### Edge Cases

- 利用統計が全くない新規ユーザーには、どの順序で表示するか？（既定順）
- すべてのおすすめソースを追加済みの場合、リストは空になる（feature 005 と同じ挙動）
- 利用傾向が拮抗している（共有と興味なしが同程度）の場合、スコアが同点になり、静的リスト順で安定して並ぶ（FR-009）
- 利用統計が更新された後、いつ並び順に反映されるか（シート再表示時か、リアルタイムか）？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはユーザーの記事への反応履歴（共有・興味なし）をキーワード別に集計し、各おすすめソースの代表キーワードと一致するシグナルの重み（共有+2/興味なし-3）を合算して優先度スコアを算出しなければならない
- **FR-002**: システムはおすすめソースを優先度スコアの高い順に並び替えて表示しなければならない
- **FR-003**: 各おすすめソースは、利用シグナルのキーワードと照合可能な代表キーワード群を持たなければならない
- **FR-009**: スコアが同点のおすすめソースは、現行の静的リスト順（コード定義順）を維持して並べなければならない
- **FR-004**: 利用統計が存在しない場合、システムはおすすめを既定の順序（現行の静的順）で表示しなければならない
- **FR-005**: システムは追加済みソースをおすすめリストから除外しなければならない（feature 005 の挙動を維持）
- **FR-006**: 並び替えはローカルに蓄積されたデータのみを用いて行い、リモート通信や手動編集を伴ってはならない
- **FR-007**: システムはおすすめソース全件（現行7件）を並び替えて表示し、件数の絞り込み（一部非表示）は行わない
- **FR-008**: システムはソース追加シートを開くたびに優先度スコアを再計算し、最新の利用傾向を並び順に反映しなければならない

### Key Entities

- **おすすめソース (SuggestedSource)**: 推奨される未追加ソース。名前・フィードURL に加え、利用シグナルと照合するための代表キーワード群を持つ
- **利用シグナル (InterestSignal)**: 既存エンティティ。記事への共有・興味なしの反応とキーワード・ソース名・重みを保持。優先度スコア算出の入力となる
- **優先度スコア**: 各おすすめソースに対し、利用シグナルとの照合で動的に算出される並び替え用の数値（永続化は不要、表示時に計算）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 特定ジャンルへの肯定的な反応（共有）を蓄積したユーザーには、対応するおすすめソースが上位（上位3位以内）に表示される
- **SC-002**: 利用履歴のない新規ユーザーには、おすすめが既定順で表示され、空や不正な並びにならない（再現率 100%）
- **SC-003**: ソース追加シートを開いてからおすすめリストが表示されるまでに、体感できる遅延が生じない
- **SC-004**: 追加済みソースがおすすめに表示される誤りが発生しない（再現率 0%、feature 005 の維持）

## Assumptions

- 並び替えの根拠には既存の `InterestSignal`（共有 +2 / 興味なし -3 の重み）を再利用する
- おすすめソースとユーザー傾向の照合は、おすすめソースに付与した代表キーワード群と、シグナルのキーワードを突き合わせて行う（Clarifications Q1）
- 同点時は静的リスト順を維持する（Clarifications Q2 / FR-009）
- 優先度スコアは永続化せず、シートを開くたびに都度算出する（FR-008、パーソナライズの既存方針と整合）
- おすすめは全件を並び替えて表示し、件数の絞り込みは行わない（FR-007）
- おすすめソースのカタログ自体は引き続き静的（7件、コード内定義）。本フィーチャーは「並び順・優先度」のみを動的化する
- リモート取得・手動編集は対象外（ユーザー指定の制約）
