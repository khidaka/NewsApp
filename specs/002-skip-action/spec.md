# Feature Specification: スキップアクション

**Feature Branch**: `002-skip-action`

**Created**: 2026-05-29

**Status**: Draft

**Input**: 興味がないわけではないが今回はスキップする、という判断を行えるようにして欲しい。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 記事をスキップして後回しにする (Priority: P1)

ユーザーは記事を読んで「興味はあるが今は読む時間がない」「後でまた見たい」と感じたとき、「興味なし」に分類せずそのセッションだけ記事をスタックから外すことができる。スキップした記事はパーソナライズアルゴリズムに影響せず、次回アプリを開いたときに再びスタックに表示される可能性がある。

**Why this priority**: 「興味なし」との使い分けが本機能の核心。これだけで完結するため P1。

**Independent Test**: 記事をスキップし、アプリを再起動すると同じ記事が再びスタックに現れることを確認できる。

**Acceptance Scenarios**:

1. **Given** ニュースカードが表示されている状態で、**When** ユーザーがスキップ操作をする、**Then** そのカードがスタックから消え「興味なし」も「共有済み」も記録されない
2. **Given** 記事をスキップした状態で、**When** アプリを再起動して新しいニュースを取得する、**Then** スキップした記事がスタックに再表示される候補となる（7日以内かつ未スワイプとして扱われる）
3. **Given** 記事をスキップした状態で、**When** 同じセッション中にリロードする、**Then** スキップした記事は再表示されない（同セッション内は除外）
4. **Given** スキップ操作の直後に、**When** ユーザーが Undo ボタンをタップする、**Then** スキップした記事がスタック先頭に戻る

---

### User Story 2 - スキップ操作の UI (Priority: P2)

ユーザーは既存の左右スワイプを邪魔せずに、直感的な操作でスキップを実行できる。

**Why this priority**: P1 の動作が固まった後に UI を洗練させる。操作方法の選択肢が複数あるため P2 として分離。

**Independent Test**: スキップ専用の操作でカードが消え、右スワイプ・左スワイプの動作が従来通り機能することを確認できる。

**Acceptance Scenarios**:

1. **Given** ニュースカードが表示されている状態で、**When** ユーザーが上スワイプする、**Then** スキップアクションが実行される
2. **Given** ニュースカードが表示されている状態で、**When** カード上のスキップボタンをタップする、**Then** スキップアクションが実行される
3. **Given** スキップ操作を実行した直後に、**When** 画面を確認する、**Then** 次のカードが表示され「右スワイプ = 共有」「左スワイプ = 興味なし」の動作は変わっていない

---

### Edge Cases

- スキップした記事が50件上限の次回フェッチ時に上位50件に選ばれなかった場合、再表示されないことがある（仕様通り）
- すべての記事をスキップしてスタックが空になった場合、空状態のガイダンスを表示する
- スキップした記事を Undo した後にすぐ別の操作をした場合、Undo できるのは直前の1件のみ（既存仕様と同じ）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ユーザーはスキップ操作（上スワイプまたは専用ボタン）によって記事をスタックから除外できなければならない
- **FR-002**: スキップした記事は「共有」「興味なし」のいずれにも分類されてはならず、パーソナライズアルゴリズムに影響を与えてはならない
- **FR-003**: スキップした記事はそのセッション（アプリ終了まで）の間、スタックに再表示されてはならない
- **FR-004**: スキップした記事はアプリ再起動後の次回ニュース取得において、未スワイプの記事として扱われ再表示される候補とならなければならない
- **FR-005**: スキップ操作は既存の Undo 機能（直前の1件取り消し）の対象でなければならない
- **FR-006**: スキップ操作を実行しても、右スワイプ（共有）および左スワイプ（興味なし）の動作は変わってはならない

### Key Entities

- **Article（記事）**: 既存エンティティに `skippedInSession` フラグ（セッション内一時フラグ、永続化なし）を追加
- スキップ状態は永続化しない（`swipeAction` には記録しない）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: スキップ操作からカードが消えるまで0.5秒以内である
- **SC-002**: スキップした記事を含むセッション内でリロードしても、スキップ済み記事が再表示されない
- **SC-003**: アプリ再起動後、スキップした記事が（50件上限に収まる範囲で）次回スタックに出現する
- **SC-004**: スキップ後に Undo で記事がスタック先頭に戻る

## Assumptions

- スキップはセッションスコープの概念であり、永続化しない（アプリ再起動で状態がリセットされる）
- 上スワイプをスキップに割り当てる（下スワイプは iOS のスクロール等と競合するため使用しない）
- スキップボタンはカード上部または画面に常時表示する（デザインは実装フェーズで決定）
- スキップした記事はパーソナライズスコアが変化しないため、次回は同程度のスコアで表示される
- 既存の Undo は「直前の1件」のみのため、スキップも同じルールに従う
