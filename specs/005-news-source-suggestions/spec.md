# Feature Specification: おすすめニュースソースサジェスト

**Feature Branch**: `005-news-source-suggestions`

**Created**: 2026-05-29

**Status**: Draft

**Input**: User description: "ニュースソースの追加時に、おすすめのニュースソースをサジェストする機能が欲しいです。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - おすすめリストからワンタップ追加 (Priority: P1)

ユーザーがニュースソース追加画面を開いたとき、あらかじめ用意されたおすすめソース一覧が表示される。ユーザーはリスト内のソースをタップするだけで URL や名前を手入力することなく即座に追加できる。

**Why this priority**: 手入力の手間を省き、初回セットアップを大幅に効率化する。アプリの第一印象となる体験を改善する最重要ストーリー。

**Independent Test**: ソース追加シートを開いたときにおすすめリストが表示され、タップで追加されることを単独で確認できる。

**Acceptance Scenarios**:

1. **Given** ソース追加シートが開いている状態で、**When** おすすめリストのソースをタップしたとき、**Then** そのソースが即座にニュースソース一覧に追加される
2. **Given** ソース追加シートが開いている状態で、**When** おすすめリストのソースをタップしたとき、**Then** そのソースはおすすめリストから即座に消え、シートは開いたまま次の追加ができる
3. **Given** ソースが1件も登録されていない状態で、**When** ソース追加シートを開いたとき、**Then** おすすめリストに1件以上のソースが表示される

---

### User Story 2 - 追加済みソースの重複防止 (Priority: P2)

既に追加済みのおすすめソースはサジェストリストに表示されない。ユーザーは重複追加を気にせずリストを操作できる。

**Why this priority**: 重複追加エラーを未然に防ぎ、リストを整理された状態に保つ。既存の一意制約エラーをユーザーが意識しなくて済む。

**Independent Test**: 登録済みソースがある状態でおすすめリストを開き、既登録ソースが表示されないことを確認できる。

**Acceptance Scenarios**:

1. **Given** おすすめリスト内のソースが既に追加済みの状態で、**When** ソース追加シートを開いたとき、**Then** そのソースはおすすめリストに表示されない
2. **Given** おすすめリストの全ソースが追加済みの状態で、**When** ソース追加シートを開いたとき、**Then** おすすめセクション自体が非表示になる

---

### User Story 3 - 手動 URL 入力との共存 (Priority: P3)

おすすめリストにないソースを追加したいとき、引き続き従来の URL 手入力フォームを使用できる。サジェスト機能はあくまで補助であり、既存フローを妨げない。

**Why this priority**: パワーユーザーや独自ソースを追加したいケースへの対応。既存機能の後退を防ぐ。

**Independent Test**: おすすめリストが表示されつつ、URL 手入力フォームも引き続き利用可能であることを確認できる。

**Acceptance Scenarios**:

1. **Given** ソース追加シートが開いている状態で、**When** ユーザーが URL フィールドに直接入力して「追加」をタップしたとき、**Then** 従来通りソースが追加される

---

### Edge Cases

- おすすめリストの全ソースが追加済みの場合、おすすめセクションは非表示にする
- おすすめリストが空（定義ゼロ件）の場合、セクション自体を非表示にする
- URL 末尾スラッシュや大文字小文字の差異による重複チェック漏れは V1 スコープ外（完全一致で判定）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: ソース追加シート内の上部におすすめニュースソースの一覧セクションが表示され、手動 URL 入力フォームはその下に続かなければならない
- **FR-002**: おすすめリストはアプリにバンドルされた静的リストであり、ネットワーク取得は不要
- **FR-003**: 各おすすめソースには表示名とフィード URL が含まれなければならない
- **FR-004**: ユーザーがおすすめソースをタップしたとき、そのソースは即座に追加され、シートは開いたまま連続追加できなければならない
- **FR-005**: 既に追加済みのおすすめソースはリストに表示してはならない
- **FR-006**: おすすめリスト内の全ソースが追加済みの場合、おすすめセクションは表示されない
- **FR-007**: 既存の URL 手入力フォームはおすすめ機能追加後も引き続き動作しなければならない
- **FR-008**: おすすめリストには最低5件のニュースソースが含まれなければならない

### Key Entities

- **SuggestedSource**: おすすめとして提示されるニュースソース定義。表示名・フィード URL を持ち、アプリにバンドルされる。
- **NewsSource**: 既存のニュースソースモデル（`feedURL`, `name`, `isEnabled`, `addedAt`）。おすすめからの追加でも同一モデルを使用。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ソース追加シートを開いてからおすすめリストをタップするまで2ステップ以内でソースを追加できる
- **SC-002**: おすすめリストには最低5件のニュースソースが含まれる
- **SC-003**: 既登録ソースがおすすめリストに表示される件数はゼロ（重複表示率 0%）
- **SC-004**: おすすめからの追加操作後、画面更新が1秒以内に視覚的に確認できる

## Clarifications

### Session 2026-05-29

- Q: ソース追加シート内でおすすめリストをどこに配置するか → A: 上部におすすめリスト、下部に手動 URL フォーム（候補を先に見せ、見当たらなければ手入力）
- Q: おすすめソースをタップして追加した後、シートはどうなるか → A: シートは開いたまま（タップしたソースがリストから消え、続けて複数追加できる）

## Assumptions

- おすすめソースリストはアプリコード内に静的に定義される（外部 API・サーバー管理は不要）
- おすすめソースは日本語ニュースを中心としたラインナップとする（ユーザーが日本語ニュース読者のため）
- カテゴリ分類（例: 一般ニュース、テック、海外）は V1 では任意扱いとし、実装者の判断に委ねる
- URL の重複判定は完全一致とし、正規化（末尾スラッシュ・大文字小文字）は V1 スコープ外
- アプリの既存 `NewsSource` モデルをそのまま使用し、新たなデータモデルは追加しない
